# Development Log 001 — Phase-One Foundation

- **Log ID:** `LOG_001_1783049404`
- **Date:** 3 July 2026
- **Status:** Foundation implemented and locally verified
- **Source plan:** [`docs/MAIN_PLAN.md`](../MAIN_PLAN.md)

## Summary

The repository has progressed from a clean slate to a working phase-one application foundation for multitenant restaurant operations checklists. The implementation follows a server-rendered, progressively enhanced architecture using Fastify, Nunjucks, TypeScript, Prisma, PostgreSQL, and Zod.

The current milestone establishes the core security boundaries, domain model, hard-coded form system, responsive UI, and offline-first browser behavior. Provider-dependent features such as private S3 uploads and Postmark reporting are intentionally left as explicit integration seams rather than incomplete public endpoints.

## Application foundation

- Added strict TypeScript configuration with separate development typechecking and production build targets.
- Added Fastify 5 application and server entry points.
- Added Nunjucks server-rendered views and reusable form field macros.
- Added centralized error handling with separate HTML and API responses.
- Added graceful `SIGINT` and `SIGTERM` shutdown handling.
- Added a lightweight health endpoint at `GET /health`.
- Upgraded the runtime target to Node.js 20.19 or newer; Docker uses Node.js 22.

## Configuration

- Added `__main_config.json` for non-secret application, upload, security, reporting, and tenant settings.
- Added `.env.example` for deployment-sensitive values.
- Added Zod schemas for both runtime environment variables and main application configuration.
- Configuration is validated at startup and invalid values fail fast.
- Tenant palettes accept only six-digit hexadecimal colors, and configured asset paths must be application-relative.
- UTC remains the storage and application standard, with a reusable UTC+4 display formatter.

## Multitenancy and tenant isolation

- Added hostname-based tenant resolution middleware.
- Tenant identity must exist in both validated configuration and the database; an unprovisioned tenant returns `503` instead of silently bypassing tenant identity.
- Added request-scoped tenant metadata for branding and authorization.
- Added a `withTenant` database helper that opens a transaction and sets the transaction-local PostgreSQL variable `app.current_tenant_id` before tenant-owned ORM operations.
- Added explicit PostgreSQL RLS policies for all tenant-owned tables in `prisma/rls.sql`.
- RLS is both enabled and forced in the supplied SQL.
- Added a database trigger that prevents updates or deletion of audit events.

Deployment must use an application database role that does not own the protected tables and does not have `BYPASSRLS`; otherwise PostgreSQL may bypass the policies.

## Authentication and authorization

- Added login and logout routes.
- Added Argon2 password hashing and verification helpers.
- Added secure, HTTP-only, same-site session cookies with configurable expiry.
- Session identifiers are regenerated after successful login.
- Added role definitions for Super Admin, Admin, Manager, Contributor, Viewer, and Auditor.
- Added reusable role guards for form reading and form completion.
- Added login rate limiting and a configurable application-wide rate limit.

The default Fastify in-memory session store is suitable only for local development. A PostgreSQL or Redis-backed store remains required before production deployment or horizontal scaling.

## Database model

Added a Prisma schema covering:

- Tenants and users.
- User role assignments.
- Form settings.
- Form submissions and lifecycle state.
- Submission files and retention metadata.
- Submitter and verifier signatures.
- Verification assignments.
- Append-only audit events.
- Report jobs and report status.

Important persistence behavior includes:

- UUID identifiers for tenant-owned entities.
- UTC timestamp columns using PostgreSQL `timestamptz`.
- JSON submission values alongside normalized metadata.
- Tenant-scoped idempotency-key uniqueness.
- Form-key and creation-time indexes for submission lookup.
- Upload checksum, size, generated filename, expiration, and deletion metadata.

No deployment-specific baseline migration has been committed yet. The Prisma base migration should be generated and reviewed for the target database, then `prisma/rls.sql` should be applied after the generated table DDL.

## Hard-coded form system

- Added a typed, versioned form registry designed to remain compatible with a future dynamic builder.
- Added the first form: **Daily Opening Checklist**, version 1.
- Added reusable field types for text, textarea, radio, select, checkbox, photo, and signature inputs.
- Added server-side Zod validation generated from each form definition.
- Added validation for submission envelopes, form versions, GPS coordinates, device metadata, and idempotency keys.
- Added configurable GPS failure behavior at the form-definition level.
- Added a central state machine supporting:
  - `DRAFT -> SUBMITTED`
  - `DRAFT -> PENDING_VERIFICATION`
  - `PENDING_VERIFICATION -> SUBMITTED`
- Invalid transitions, including edits to submitted forms, are rejected.
- Every newly persisted submission records an audit event in the same tenant transaction.
- Duplicate offline requests return the canonical submission associated with the original idempotency key.

## Server-rendered interface

- Added shared application and tenant navigation headers.
- Added tenant-specific CSS variables and branding assets.
- Added login, dashboard, form list, new form, and reusable error templates.
- Added reusable Nunjucks macros for form controls.
- Added a mobile-first visual system using only the system font stack.
- Added the four responsive bands specified by the main plan: mobile portrait, tablet portrait, tablet landscape, and desktop.
- Added larger touch targets and a restrained layout intended for restaurant-floor use.

## Offline and PWA behavior

- Added a web app manifest and service worker.
- Added application-shell and visited-form caching.
- Added IndexedDB stores for local drafts and queued submissions.
- Added local draft saving from the form interface.
- Added offline submission queueing with cryptographically generated idempotency keys.

- Added user-visible sync status messages.
- Added browser GPS capture with accuracy metadata.
- Added a canvas-based signature control with clear and export behavior.
- Added device user-agent and client timestamp metadata to submission envelopes.

Photo selection is represented in the form schema and UI, but upload blobs are not yet submitted or queued. Private S3 presigning, actual file-signature validation, completion callbacks, and retention cleanup must be implemented together before uploads are enabled end to end.

## Known limitations and next developments

The following work remains before the application can be considered production-ready:

1. Generate and review the initial Prisma migration, apply RLS SQL, and verify isolation using a non-owner application database role.
2. Add controlled tenant and initial-administrator provisioning.
3. Replace the in-memory session store with PostgreSQL or Redis storage.
4. Add CSRF protection for cookie-authenticated mutations.
5. Implement draft persistence on the server and the verification assignment/review routes.
6. Implement private S3 upload presigning, content validation, completion, signed viewing URLs, and cleanup jobs.
7. Implement Admin dashboard CRUD for users, roles, form settings, verification rules, branding, recipients, and audit viewing.
8. Implement Postmark email delivery, weekly/monthly scheduling, CSV generation, and report-job claiming.
9. Add database integration tests for RLS, idempotency, audit immutability, and concurrent submission behavior.
10. Add Playwright coverage for login, mobile completion, offline queueing, reconnection sync, and role restrictions.
